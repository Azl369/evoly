# Evoly V0.1 原型版研发计划

## 1. 版本目标

V0.1 原型版的目标是验证 Evoly 的最小核心闭环：

> 用户能创建目标和任务，能查看今日任务，并能标记任务完成。

V0.1 不追求完整商业化体验，不追求复杂提醒、统计、复盘、AI Coach 或多端同步。重点是让用户第一次打开软件时，可以完成一条最短路径：

```text
创建目标 → 添加任务 → 在今日页看到任务 → 标记完成 → 看到完成反馈
```

## 2. 版本范围

### 2.1 必须包含

- Windows 桌面端可运行。
- 本地 SQLite 数据持久化。
- 目标创建。
- 目标列表查看。
- 目标详情查看。
- 子任务创建。
- 今日任务查看。
- 任务完成。
- 基础进度展示。
- 基础空状态、加载状态、错误状态。

### 2.2 暂不包含

- 系统级提醒。
- 重复任务。
- 周/月统计。
- 目标复盘。
- AI Coach。
- 多端同步。
- 账号系统。
- 数据导入导出。
- 复杂日历视图。
- 后台常驻托盘。

说明：当前代码中已经存在部分 V0.2/V0.3 能力，例如提醒、统计、筛选排序等。V0.1 研发计划仍以“原型版验收最小闭环”为准，后续能力可保留，但不作为 V0.1 验收阻塞项。

## 3. 目标用户场景

### 3.1 第一次使用

用户打开 Evoly 后，可以：

1. 进入目标页。
2. 创建一个目标。
3. 为目标添加今天的第一步任务。
4. 回到今日页。
5. 看到刚创建的任务。
6. 点击完成任务。
7. 看到任务被划掉或标记完成。

### 3.2 日常使用

用户每天打开 Evoly 后，可以：

1. 在今日页看到今天需要完成的任务。
2. 按优先级理解任务重要性。
3. 完成任务后快速标记。
4. 在目标详情里看到目标推进情况。

## 7. 非功能要求

### 7.1 性能

- 应用启动速度要可接受。
- 今日页和目标列表页滚动不能明显卡顿。
- 完成任务后的 UI 反馈应即时出现。
- 数据库查询不能阻塞主线程造成明显卡顿。

### 7.2 可靠性

- 目标和任务不能因应用重启丢失。
- 数据库初始化失败时应显示错误状态。
- 任务完成保存失败时应回滚 UI 状态或给出提示。

### 7.3 体验

- 创建目标流程要短。
- 今日页要直接回答“今天做什么”。
- 完成态要有明显视觉反馈，例如勾选、划线、进度变化。
- 空状态文案要指导用户下一步做什么。

## 9. 验收用例

### 用例 1：创建目标和任务

步骤：

1. 打开 Evoly。
2. 进入目标页。
3. 点击新建目标。
4. 输入目标名称。
5. 输入今天第一步任务。
6. 点击创建。

预期：

- 目标创建成功。
- 目标出现在目标列表。
- 第一条任务写入数据库。

### 用例 2：查看今日任务

步骤：

1. 创建一个今天到期的任务。
2. 返回今日页。

预期：

- 今日页展示该任务。
- 任务显示优先级和预计耗时。

### 用例 3：完成任务

步骤：

1. 在今日页点击任务完成按钮。

预期：

- 任务变为完成态。
- 任务标题出现删除线或勾选状态。
- 今日进度条更新。
- 重启应用后任务仍保持完成状态。

### 用例 4：查看目标进度

步骤：

1. 创建一个目标和多个任务。
2. 完成其中一个任务。
3. 进入目标详情页。

预期：

- 目标进度按子任务完成比例更新。
- 已完成子任务显示完成态。

### 用例 5：空状态

步骤：

1. 在没有今日任务的情况下打开今日页。

预期：

- 显示空状态。
- 文案引导用户去目标页创建任务。

## 10. 风险与应对

### 10.1 SQLite 迁移风险

风险：

- 数据表结构变化导致旧数据不可用。

应对：

- V0.1 保持数据模型简单。
- 每次升级数据库版本都写迁移逻辑。

### 10.2 页面状态复杂度上升

风险：

- 页面同时处理加载、错误、空数据、编辑状态，容易混乱。

应对：

- 页面状态先保持简单。
- V0.1 不引入复杂全局状态管理。
- Repository 作为数据边界。

### 10.3 功能过度扩张

风险：

- 原型阶段加入提醒、统计、Coach 等功能，导致主链路不稳定。

应对：

- V0.1 验收只看目标、任务、今日、完成闭环。
- 超出范围的功能不作为 V0.1 阻塞项。
