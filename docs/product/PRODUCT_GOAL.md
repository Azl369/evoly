# Evoly：个人成长追踪与目标驱动产品定义

## 1. 产品定位

Evoly 是一款面向个人用户的成长追踪与目标驱动工具，核心价值是帮助用户把长期目标拆解成可执行任务，并通过提醒、进度反馈和复盘机制，持续督促用户完成目标。

产品不只是一个待办清单，而是一个“目标推进系统”：它关注目标从设定、拆解、执行、提醒、追踪到复盘的完整闭环。

## 2. 产品愿景

让用户每天都清楚自己该做什么、为什么做、做到哪里了，并在拖延时得到及时提醒和推动。

## 3. 核心目标

- 帮助用户把模糊目标转化为具体、可执行、可追踪的计划。
- 通过时间提醒、周期提醒和延迟提醒，降低遗忘和拖延概率。
- 用进度可视化、完成记录和阶段复盘，增强用户的持续行动动力。
- 让目标管理过程尽量轻量，避免复杂配置造成使用负担。

## 4. 目标用户

- 学生：用于学习计划、考试备考、作业安排、习惯培养。
- 上班族：用于工作任务、项目节点、个人成长目标管理。
- 自律提升用户：用于健身、读书、早睡、技能学习等长期目标。
- 自由职业者：用于多个项目和个人事务的时间安排。

## 5. 典型使用场景

- 用户设定“30 天学完 Python 基础”，软件帮助拆解为每日学习任务。
- 用户创建“每天 22:30 睡觉”的习惯目标，软件每天提醒并记录完成情况。
- 用户有一个工作项目，需要设置多个截止日期和阶段性提醒。
- 用户连续几天没有推进某个目标，软件提示用户重新安排任务或降低目标难度。

## 6. MVP 功能范围

第一版应优先完成最小可用闭环：创建目标、拆解任务、设置提醒、记录完成、查看进度。

### 6.1 目标管理

- 创建目标：填写目标名称、目标描述、截止日期、重要程度。
- 设置目标类型：一次性目标、周期性目标、长期目标。
- 设置目标状态：未开始、进行中、已完成、已暂停、已放弃。
- 查看目标列表：按截止时间、重要程度、完成状态筛选。
- 编辑与删除目标。

### 6.2 任务拆解

- 在目标下创建多个子任务。
- 子任务支持标题、说明、截止时间、预计耗时、优先级。
- 子任务支持完成、延期、取消。
- 支持把长期目标拆成阶段任务，例如“第 1 周”“第 2 周”“最后复盘”。

### 6.3 提醒机制

- 单次提醒：指定日期和时间提醒用户执行任务。
- 重复提醒：支持每天、每周、每月、自定义间隔。
- 截止提醒：在目标或任务截止前提醒，例如提前 1 天、提前 1 小时。
- 未完成提醒：任务到期未完成时再次提醒用户处理。
- 今日任务提醒：每天固定时间推送当天待办。

### 6.4 今日计划

- 展示今天需要完成的目标和任务。
- 支持快速标记完成。
- 支持把任务延后到明天或指定日期。
- 支持按优先级展示“今天最重要的 3 件事”。

### 6.5 进度追踪

- 目标进度根据子任务完成比例自动计算。
- 显示连续完成天数、延期次数、目标完成率。
- 支持查看本周、本月完成情况。
- 支持简单图表展示完成趋势。

### 6.6 复盘与反馈

- 目标完成后引导用户填写复盘：完成效果、遇到的问题、下次改进。
- 目标延期时提示用户选择原因：时间不足、目标过大、忘记执行、优先级变化。
- 根据延期原因给出建议，例如拆小任务、降低频率、重新设置截止日期。

### 6.7 目标档案夹与文档库

- 每个目标天然对应一个目标档案夹。
- 目标档案夹中沉淀该目标涉及的过程记录、问题记录、复盘、项目总结和可复用知识。
- 文档库首页优先展示目标档案夹，而不是按文档类型组织内容。
- 文档类型只作为档案夹内标签，不作为一级导航。
- 未关联目标的文档进入“未归档文档”，后续可再关联到目标。
- 目标完成后支持从目标档案夹中生成结构化项目总结。

## 7. 非 MVP 功能

以下功能可作为后续版本规划，不建议第一版同时实现：

- AI 自动拆解目标。
- 日历同步。
- 多端同步。
- 团队协作。
- 社交打卡。
- 成就系统和积分商城。
- 语音输入任务。
- 智能推荐最佳执行时间。
- 目标档案夹式 Markdown 文档库。
- 项目完成后生成结构化总结文档。

### P0：必须完成

- 今日任务列表。

### P1：重要增强

### P2：后续优化

## 9. 核心页面定义

### 9.1 首页 / 今日页

- 展示今天要做的任务。
- 展示即将截止的目标。
- 展示今天最重要的任务。
- 提供快速完成、延期、查看详情入口。

### 9.2 目标列表页

- 展示所有目标。
- 支持按状态、截止日期、优先级筛选。
- 支持新建目标。

### 9.3 目标详情页

- 展示目标说明、截止日期、完成进度。
- 展示该目标下的子任务列表。
- 支持添加子任务、设置提醒、填写复盘。
- 支持查看该目标的档案夹入口和最近文档。
- 支持在该目标档案夹中新建 Markdown 文档。
- 目标完成后支持创建项目总结模板。

### 9.4 统计页

- 展示目标完成率。
- 展示任务完成数量。
- 展示连续完成天数。
- 展示延期任务数量。

### 9.5 设置页

- 设置默认提醒时间。
- 设置提醒方式。
- 设置每日报告时间。
- 管理数据备份与导出。

## 10. 数据对象定义

### 10.1 Goal 目标

- id：唯一标识。
- title：目标名称。
- description：目标描述。
- type：目标类型。
- priority：优先级。
- status：目标状态。
- startDate：开始日期。
- dueDate：截止日期。
- progress：完成进度。
- createdAt：创建时间。
- updatedAt：更新时间。

### 10.2 Task 子任务

- id：唯一标识。
- goalId：所属目标。
- title：任务名称。
- description：任务说明。
- priority：优先级。
- status：任务状态。
- estimatedMinutes：预计耗时。
- dueDateTime：截止时间。
- completedAt：完成时间。

### 10.3 Reminder 提醒

- id：唯一标识。
- targetType：提醒对象类型，目标或任务。
- targetId：提醒对象 ID。
- remindAt：提醒时间。
- repeatRule：重复规则。
- enabled：是否启用。

### 10.4 Review 复盘

- id：唯一标识。
- goalId：所属目标。
- result：完成结果。
- problem：遇到的问题。
- improvement：改进措施。
- createdAt：复盘时间。

## 11. 用户体验原则

- 创建目标要简单，避免让用户填写过多字段。
- 首页必须直接回答“我今天要做什么”。
- 提醒要有帮助，但不能过度打扰。
- 延期不应只惩罚用户，而应帮助用户重新安排。
- 统计要鼓励用户继续行动，而不是制造挫败感。

## 12. 成功指标

- 用户每天打开软件的比例。
- 用户创建目标后的任务拆解比例。
- 今日任务完成率。
- 目标按期完成率。
- 用户连续使用天数。
- 延期任务被重新安排的比例。

## 13. 第一阶段开发建议

第一阶段建议先做本地单机版本，不急于实现账号系统和云同步。优先保证目标、任务、提醒、进度这条主链路稳定可用。

建议开发顺序：

1. 搭建基础项目结构。
2. 实现目标与任务的数据模型。
3. 实现目标列表、今日任务、目标详情页面。
4. 实现任务完成与进度计算。
5. 实现本地提醒。
6. 实现基础统计和复盘。

## 15. 产品一句话定义

一款帮助用户把目标拆成每日行动，并通过提醒、追踪和复盘持续推动完成的个人目标管理软件。

## 16. Evoly Coach 智能成长助理规划

随着产品定位从“任务提醒与目标安排”升级为“个人成长追踪与目标驱动”，Evoly 可以在后续版本中内嵌一个智能 Agent，用来帮助用户更好地管理目标、任务、提醒和复盘。

但 Evoly 不应被设计成一个泛聊天软件。更合理的定位是：

> Evoly 是个人成长系统，Coach 是其中的智能助理层。

Agent 的职责不是替代用户管理所有事项，而是在复杂、模糊、容易拖延的场景中，帮助用户做目标拆解、计划调整、拖延干预和复盘总结。

### 16.1 功能定位

Evoly Coach 应作为产品的“成长助理层”，而不是主流程本身。

基础目标、任务、提醒、统计功能必须在没有 Agent 的情况下仍然完整可用。Agent 只在用户需要帮助时出现，提供建议、草案和分析，最终是否执行由用户确认。

核心定位：

- 帮用户把模糊目标拆成可执行计划。
- 帮用户从过多任务中筛选今日重点。
- 帮用户发现拖延、停滞和目标过载问题。
- 帮用户根据历史完成情况调整节奏。
- 帮用户生成复盘和下一步建议。

### 16.2 可行功能方向

#### 16.2.1 目标拆解

用户输入一个较大的目标后，Coach 可以协助拆解。

示例：

- “三个月提升英语口语。”
- “30 天减脂 3 公斤。”
- “两个月学完 Flutter 基础。”

Coach 可以输出：

- 目标是否过大或过于模糊。
- 阶段目标。
- 每周计划。
- 每日任务。
- 建议提醒时间。
- 建议任务难度。

这些结果可以转化为 Evoly 现有的 Goal、Task、Reminder 数据，而不是停留在一段文字建议中。

#### 16.2.2 今日任务优化

当用户今日任务过多时，Coach 可以帮助用户重新安排。

可提供能力：

- 选出今天最重要的 3 个任务。
- 判断哪些任务可以延期。
- 判断哪些任务应该拆小。
- 判断哪些任务可以取消或暂停。
- 估算今日任务总耗时。

示例建议：

> 你今天有 8 个任务，预计总耗时约 5 小时。建议保留 3 个高优先级任务，其余延期到明天。

#### 16.2.3 拖延干预

当某个目标连续多天没有推进，或任务反复延期时，Coach 可以主动给出温和干预。

可识别的问题：

- 目标太大。
- 任务启动成本太高。
- 提醒时间不合适。
- 用户近期精力不足。
- 目标优先级已经变化。

可给出的建议：

- 把任务缩小到 15 分钟。
- 调整提醒时间。
- 暂停低优先级目标。
- 把长期目标拆成阶段任务。
- 降低执行频率。

#### 16.2.4 复盘总结

Coach 可以根据用户完成记录生成复盘。

复盘类型：

- 今日复盘。
- 本周复盘。
- 本月复盘。
- 单个目标完成复盘。
- 目标失败或放弃复盘。

复盘内容：

- 完成了什么。
- 哪些目标推进稳定。
- 哪些任务延期较多。
- 用户高效时间段。
- 拖延原因推测。
- 下周行动建议。

示例：

> 本周你完成了 12 个任务，主要集中在晚上 9 点后。Flutter 学习目标推进较稳定，但健身目标连续延期 3 次。建议下周把健身任务改成“饭后散步 15 分钟”，降低启动难度。

#### 16.2.5 对话式管理

用户可以通过自然语言管理目标和任务。

示例指令：

- “帮我把 Flutter 学习目标重新安排一下。”
- “今天太累了，把不重要的任务延期。”
- “我想开始健身，帮我建一个 30 天计划。”
- “最近总是拖延，帮我分析原因。”

Coach 应将这些指令转换成可确认的操作草案，例如：

- 新建目标。
- 新建任务。
- 修改优先级。
- 延期任务。
- 暂停目标。
- 生成复盘。

所有实际数据修改都应由用户确认后执行。

### 16.3 避免产品臃肿的原则

Agent 功能有价值，但如果设计不当，会让产品变得臃肿。

应避免：

- 首页塞入大型聊天窗口。
- 所有操作都强制经过 Agent。
- 每个页面都放过多 AI 按钮。
- 用户创建普通任务也必须等待 Agent。
- Agent 输出大量用户不需要的建议。
- 把 Evoly 做成泛聊天软件。

应坚持：

- Agent 是辅助层，不是主流程。
- 基础任务管理不依赖 Agent。
- 复杂场景才主动出现 Coach。
- Coach 建议必须可以一键采纳或忽略。
- Coach 输出必须能落到目标、任务、提醒、复盘等已有模块。
- Coach 不做泛聊天，只做成长管理相关能力。

### 16.4 推荐产品形态

不建议把该能力命名为“AI 聊天助手”。更推荐使用：

> Evoly Coach

原因：

- “Coach”更符合个人成长、目标推进和持续改进的语义。
- 用户需要的是陪伴式成长助理，而不是一个泛聊天机器人。
- 该名称能强化 Evoly 的成长驱动定位。

### 16.5 入口设计

Coach 可以分三层出现。

#### 16.5.1 轻提示

在页面中以小卡片形式出现。

示例：

> 今天任务偏多，预计 4 小时。建议保留 3 个重点任务。

可选操作：

- 接受建议。
- 稍后再说。
- 查看原因。

#### 16.5.2 局部助手

在具体页面提供上下文能力。

今日页：

- 帮我安排今天。
- 帮我延期低优先级任务。
- 帮我挑出最重要的 3 件事。

目标详情页：

- 帮我拆解目标。
- 帮我调整计划。
- 帮我分析为什么停滞。
- 帮我生成复盘。

统计页：

- 帮我分析本周表现。
- 给我下周建议。
- 生成本周复盘。

#### 16.5.3 完整 Coach 页面

可以提供一个独立的 Coach 页面，用于处理复杂指令。

但该页面不应作为首页。首页仍应优先回答：

> 我今天要做什么？

### 16.6 第一版 Coach MVP

第一版不建议直接做全能 Agent。建议先做一个轻量 MVP。

推荐 MVP 能力：

1. 目标拆解。
2. 今日任务优化。
3. 周复盘生成。

#### 16.6.1 目标拆解

用户创建目标时，可以点击：

> 帮我拆成任务

Coach 输出阶段计划和每日任务，用户确认后写入目标和任务列表。

#### 16.6.2 今日任务优化

今日页提供：

> 帮我安排今天

Coach 根据任务数量、优先级、截止时间、延期记录，给出：

- 今天必须做。
- 可以延期。
- 建议拆小。
- 建议取消。

#### 16.6.3 周复盘生成

统计页提供：

> 生成本周复盘

Coach 输出：

- 完成情况。
- 延期情况。
- 稳定推进的目标。
- 停滞目标。
- 下周建议。

### 16.7 技术实现建议

架构上可以新增模块：

```text
features/
  coach/
    domain/
    application/
    data/
    presentation/
```

核心服务：

```text
CoachService
- generateGoalPlan()
- optimizeToday()
- generateWeeklyReview()
- suggestTaskAdjustments()
```

Coach 不应直接操作数据库。推荐流程：

```text
用户输入或点击 Coach 功能
  ↓
CoachService 读取必要上下文
  ↓
生成建议或操作草案
  ↓
用户确认
  ↓
调用现有 Repository
  ↓
保存为真实目标、任务、提醒或复盘
```

这样可以保证数据安全、操作可控，也能避免 Agent 误改用户计划。

### 16.8 联网与隐私原则

如果接入大模型能力，Coach 大概率需要联网。

但 Evoly 的基础能力必须保持离线可用：

- 没有网络时，目标、任务、提醒、统计照常可用。
- 有网络时，Coach 提供智能拆解、复盘和建议。
- 网络失败时，不影响主功能。

隐私原则：

- 明确告诉用户哪些数据会用于智能建议。
- 默认只传必要上下文，不上传全部历史。
- 用户可以关闭 Coach。
- 用户可以将敏感目标标记为“不参与智能分析”。
- 所有 Coach 建议必须由用户确认后落库。

### 16.9 产品判断

Evoly 适合内嵌 Agent，但不应变成 AI 聊天软件。

合理方向是：

> Evoly 负责记录、追踪和执行；Evoly Coach 负责拆解、分析和推动。

如果坚持“Agent 是成长助理层”的边界，Evoly 不会变臃肿，反而会形成差异化竞争力。
