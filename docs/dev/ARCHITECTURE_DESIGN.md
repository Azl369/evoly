# Evoly：软件架构设计文档

## 1. 架构定位

Evoly建议采用“本地优先、前端主导、后端可扩展”的架构。第一阶段先完成稳定的单机版本，保证目标管理、任务提醒、进度追踪和复盘闭环可用；后续再扩展账号、云同步、多端协作等能力。

为了保证软件动画丝滑，架构设计必须把性能作为基础能力，而不是最后再优化。所有页面、状态更新、数据查询和动画效果都要围绕“减少阻塞、减少重绘、减少无效刷新”来设计。

## 2. 技术选型建议

### 2.1 推荐客户端技术

建议优先使用 Flutter 作为客户端技术栈。

选择原因：

- 跨平台能力强，后续可覆盖 Android、iOS、Windows、macOS、Web。
- 自带高性能渲染管线，适合实现流畅页面切换、列表动画、进度动画和微交互。
- UI 与动画能力统一，减少不同平台表现不一致的问题。
- 本地存储、通知、桌面端打包、移动端发布都有成熟方案。

如果后续明确只做 Web 版本，也可以改为 React / Vue + PWA 架构，但动画性能约束、模块分层和数据设计仍可沿用本文档。

### 2.2 推荐基础组件

- 客户端框架：Flutter。
- 本地数据库：SQLite。
- 本地配置：轻量 Key-Value 存储。
- 提醒能力：系统本地通知。
- 状态管理：单向数据流状态管理。
- 图表能力：轻量图表组件，优先选择支持局部刷新的实现。
- 后端服务：MVP 阶段暂不需要，后续可增加云同步服务。

## 3. 架构目标

- 离线可用：没有网络也能创建目标、修改任务、接收本地提醒。
- 数据可靠：任务、目标、提醒、复盘数据必须可持久化，避免丢失。
- 提醒稳定：应用关闭后仍尽量通过系统通知提醒用户。
- 动画丝滑：核心页面滚动、切换、展开、完成反馈应保持稳定帧率。
- 可扩展：后续可平滑加入账号、云同步、AI 拆解目标、日历同步。
- 易维护：业务逻辑与 UI 解耦，减少后续功能增加时的连锁修改。

## 5. 模块划分

### 5.1 App Shell 模块

负责应用启动、路由、主题、全局错误处理、全局状态初始化。

核心职责：

- 初始化数据库。
- 初始化本地通知权限。
- 加载用户设置。
- 注册全局路由。
- 提供浅色 / 深色主题。
- 管理应用生命周期。

### 5.2 Goal 目标模块

负责目标的创建、编辑、删除、状态流转和进度计算。

核心职责：

- 创建长期目标、周期目标、一次性目标。
- 维护目标状态：未开始、进行中、已完成、已暂停、已放弃。
- 根据子任务完成情况计算目标进度。
- 提供目标详情页数据。

### 5.3 Task 任务模块

负责目标下的子任务管理。

核心职责：

- 创建子任务。
- 修改任务截止时间、优先级、预计耗时。
- 标记完成、延期、取消。
- 给今日计划页提供任务列表。

### 5.4 Reminder 提醒模块

负责提醒规则、本地通知调度和过期提醒补偿。

核心职责：

- 创建单次提醒。
- 创建重复提醒。
- 创建截止前提醒。
- 应用启动时检查遗漏提醒。
- 任务完成或延期后更新提醒计划。

### 5.5 Today 今日计划模块

负责聚合今天应该完成的任务。

核心职责：

- 查询今日任务。
- 查询即将截止任务。
- 计算今天最重要的 3 件事。
- 支持快速完成、快速延期。

### 5.6 Stats 统计模块

负责完成率、连续完成天数、延期次数等统计。

核心职责：

- 统计每日完成数量。
- 统计目标完成率。
- 统计任务延期次数。
- 生成周报和月报数据。

### 5.7 Review 复盘模块

负责目标完成或延期后的复盘记录。

核心职责：

- 创建目标复盘。
- 记录失败原因。
- 记录改进策略。
- 为后续智能建议预留数据基础。

### 5.8 Documents 文档库模块

负责目标档案夹、Markdown 文档、项目总结和文档关联。

核心职责：

- 将每个目标视为一个隐式文件夹。
- 查询某个目标文件夹下的所有文档。
- 管理未归档文档。
- 创建 Markdown 文档。
- 创建项目总结文档。
- 维护文档与目标/任务的链接关系。
- 为文档库首页提供目标文件夹列表、文档数量和最近更新信息。

架构原则：

- V0.3 不做复杂自由文件夹系统。
- 目标文件夹不需要单独建表，优先由 `goals` + `document_links` 推导。
- 文档类型是文件夹内标签，不是文档库一级导航。
- 目标详情页只展示目标档案入口和最近文档，完整文档管理进入目标文件夹页。

### 5.9 Motion 动画模块

负责统一管理动画规范，避免每个页面随意实现动画导致体验不一致。

核心职责：

- 统一动画时长。
- 统一缓动曲线。
- 统一页面切换动画。
- 统一任务完成反馈动画。
- 提供可复用的动画组件。

## 8. 数据流设计

### 8.1 创建目标流程

```text
用户输入目标
  ↓
表现层校验基础字段
  ↓
应用层调用 CreateGoalUseCase
  ↓
领域层校验目标规则
  ↓
数据层写入 SQLite
  ↓
应用层刷新目标列表状态
  ↓
表现层播放创建成功反馈动画
```

### 8.2 完成任务流程

```text
用户点击完成任务
  ↓
UI 立即进入乐观更新状态
  ↓
播放完成动画
  ↓
应用层调用 CompleteTaskUseCase
  ↓
数据层更新 Task 状态
  ↓
重新计算 Goal 进度
  ↓
更新提醒计划
  ↓
写入 EventLog
```

说明：完成任务时应优先让 UI 立即反馈，不要等数据库写入完成后才播放动画。数据库写入失败时再回滚状态并提示用户。

### 8.3 提醒调度流程

```text
创建或修改任务
  ↓
生成 Reminder 规则
  ↓
保存 Reminder 到数据库
  ↓
调用系统本地通知能力
  ↓
应用启动时重新校准提醒
```

## 9. 提醒系统架构

提醒系统要同时保证“可配置”和“可靠”。

### 9.1 提醒类型

- 固定时间提醒：例如今天 20:00 提醒。
- 重复提醒：例如每天 8:00、每周一 9:00。
- 截止前提醒：例如截止前 1 小时提醒。
- 未完成补偿提醒：到期未完成后再次提醒。
- 每日计划提醒：每天固定时间推送今日任务。

### 9.2 调度策略

- 数据库保存完整提醒规则。
- 系统通知只注册近期需要触发的提醒。
- 每次应用启动、任务修改、目标修改时重新校准提醒。
- 任务完成后取消对应提醒。
- 任务延期后重新计算提醒时间。

### 9.3 异常处理

- 用户关闭通知权限时，在首页显示提醒权限提示。
- 系统通知注册失败时记录错误，不影响任务本身保存。
- 应用长时间未打开时，启动后检查过期未完成任务并提示重新安排。

## 10. 动画与性能架构

动画丝滑是Evoly的重要体验目标，必须从架构层面约束。

### 10.1 性能目标

- 普通设备上核心交互保持 60 FPS。
- 高刷新率设备上尽量适配 90 FPS / 120 FPS。
- 页面切换首帧响应尽量小于 100ms。
- 列表滚动过程中不进行重型计算。
- 任务完成动画不阻塞数据写入。
- 首页首次可交互时间控制在 2 秒以内。

### 10.2 动画原则

- 优先使用位移、透明度、缩放动画。
- 避免频繁改变会触发布局重算的属性。
- 避免在滚动过程中同时执行大量复杂动画。
- 长列表只渲染可视区域内容。
- 图表动画只在数据变化时触发，不在每次页面重建时重复播放。
- 页面切换动画保持短、轻、自然，不追求炫技。

### 10.3 Motion Token

统一定义动画参数，避免页面体验割裂。

```text
MotionDuration
- instant: 80ms
- fast: 150ms
- normal: 240ms
- slow: 360ms

MotionCurve
- standard: easeOutCubic
- emphasized: easeOutBack
- gentle: easeInOut
```

### 10.4 核心动画场景

### 首页任务完成动画

- 点击完成后任务卡片轻微缩放。
- 勾选图标快速出现。
- 卡片淡出或移动到已完成区域。
- 目标进度条同步平滑增长。

### 目标进度动画

- 进度条只在数值变化时播放。
- 使用 240ms 左右的平滑过渡。
- 避免每次页面刷新都重新从 0 播放。

### 页面切换动画

- 首页到目标详情：使用轻量滑入或淡入。
- 弹窗和底部面板：使用短距离上移动画。
- 不使用过长的 3D 翻转或复杂粒子效果。

### 列表动画

- 新增任务时只让新增项进入动画。
- 删除任务时只让删除项退出动画。
- 排序变化时使用位置过渡，不重建整个列表。

### 10.5 避免卡顿的工程约束

- UI 线程不执行大批量统计计算。
- 统计、复盘分析、未来 AI 拆解放到后台任务或独立计算单元。
- 数据库查询必须分页或限制数量。
- 首页只加载今天需要的数据，不一次性加载全部历史数据。
- 列表项组件必须尽量轻量，避免嵌套过深。
- 复杂图表按需加载，不进入首页首屏关键路径。
- 动画组件和业务组件分离，避免业务状态变化导致动画树整体重建。

### 10.6 动画质量门禁

每次新增页面或复杂动画时，需要检查：

- 滚动是否稳定。
- 页面切换是否掉帧。
- 快速连续点击是否会造成动画状态错乱。
- 低端设备是否仍可用。
- 深色模式下动画效果是否清晰。
- 开启省电模式时是否可以减少非必要动画。

## 11. 状态管理设计

### 11.1 状态类型

- 全局状态：主题、语言、通知权限、用户设置。
- 页面状态：当前筛选条件、加载状态、错误状态。
- 业务状态：目标列表、任务列表、提醒规则、统计结果。
- 临时交互状态：动画中、选中项、展开项。

### 11.2 状态原则

- 业务状态和动画状态分离。
- 数据库是最终可信数据源。
- UI 可以做乐观更新，但必须支持失败回滚。
- 页面只订阅自己需要的数据，避免全局状态变化导致整页重建。
- 列表项只订阅对应任务数据，避免一个任务变化导致整个列表刷新。

## 12. 页面架构

### 12.1 首页 / 今日页

首页是最高频页面，应保持最轻。

页面数据：

- 今日任务。
- 即将截止任务。
- 最重要的 3 件事。
- 今日完成进度。

性能要求：

- 首页不加载全部目标历史。
- 首页统计只显示轻量指标。
- 首屏骨架屏不超过 1 秒。
- 完成任务后局部刷新任务卡片和进度条。

### 12.2 目标列表页

页面数据：

- 目标摘要。
- 目标状态。
- 目标进度。
- 截止日期。

性能要求：

- 使用分页或懒加载。
- 筛选和排序结果缓存。
- 列表项避免复杂阴影和过多嵌套动画。

### 12.3 目标详情页

页面数据：

- 目标详情。
- 子任务列表。
- 目标进度。
- 提醒规则。
- 复盘入口。

性能要求：

- 子任务列表局部刷新。
- 进度动画只在完成比例变化时触发。
- 编辑弹窗与详情页状态隔离。

### 12.4 统计页

页面数据：

- 周完成情况。
- 月完成情况。
- 连续完成天数。
- 延期次数。

性能要求：

- 统计结果缓存。
- 大范围统计异步计算。
- 图表动画可关闭或降级。

### 13.1 表结构建议

```text

reminders
documents
document_links

event_logs

```

### 13.2 索引建议

```text
tasks(goalId)
tasks(dueDateTime)
tasks(status)
tasks(priority)
reminders(remindAt)
reminders(targetType, targetId)
documents(updatedAt)
documents(type, updatedAt)
document_links(documentId)

event_logs(createdAt)
```

### 13.3 目标文件夹式文档库

V0.3 文档库采用“目标即文件夹”的隐式结构：

```text
Goal
  └─ document_links(target_type = goal, target_id = goal.id)
        └─ Document
```

这种设计避免过早引入复杂文件夹表，同时能满足用户心智：

```text
一个目标 = 一个项目文件夹
文件夹里 = 该目标涉及到的过程文档、复盘、项目总结
```

查询目标文件夹内容：

```sql
SELECT d.*
FROM documents d
JOIN document_links l ON l.document_id = d.id
WHERE l.target_type = 'goal'
  AND l.target_id = :goalId
  AND d.deleted_at IS NULL
ORDER BY d.updated_at DESC;
```

文档库首页的目标文件夹列表可以由 `goals` 聚合 `document_links` 得到：

```text
目标标题
目标进度
文档数量
最近更新文档
最后更新时间
```

未归档文档定义：

```text
documents 中不存在 target_type = goal 的 document_links 记录
```

### 13.4 数据迁移

- 每次数据库结构变化必须提供迁移脚本。
- 迁移前备份关键数据。
- 迁移失败时提示用户并保留旧数据。
- 不允许直接删除用户历史数据。

## 14. 后端扩展架构

MVP 阶段不强制建设后端。后续如果需要多端同步，可增加云服务。

### 14.1 后端能力

- 用户账号。
- 数据同步。
- 数据备份。
- 多端冲突解决。
- AI 目标拆解。

### 14.2 同步策略

- 本地数据优先。
- 每条记录包含 updatedAt 和 version。
- 本地操作写入 EventLog。
- 网络恢复后增量同步。
- 冲突时优先保留用户可见的最新修改，并提供恢复入口。

## 15. 安全与隐私

- 默认本地保存用户目标和任务数据。
- 不上传数据，除非用户主动开启同步。
- 用户可导出和删除本地数据。
- 敏感设置可加密保存。
- 后续云同步需要提供登录保护和传输加密。

## 16. 错误处理策略

- 用户输入错误：在表单内即时提示。
- 数据保存失败：保留用户输入并提示重试。
- 通知权限缺失：引导用户开启权限。
- 数据库异常：记录错误日志并提示备份恢复。
- 动画异常：不影响业务操作，必要时降级为无动画。

## 19. 架构结论

第一版建议采用 Flutter + SQLite + 本地通知的本地优先架构。该架构能快速实现目标管理软件的核心闭环，同时具备较好的动画能力、离线能力和后续扩展空间。

为了保证动画丝滑，项目从一开始就应建立 Motion 动画模块、性能预算、列表懒加载、局部刷新和后台计算机制。不要等功能完成后再临时优化动画，否则很容易出现页面卡顿、状态错乱和维护困难的问题。
